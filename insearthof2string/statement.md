В рамках работы с `nav_butify` мы приняли в качестве внешнего наблюдателя текущего состояния среды результаты анализатора из [1503.json](/c:/55555/switch/Evensteam/bin/1503.json) и [my_debug_log_2026-03-15T093437.168.json](/c:/Users/upolz/Downloads/my_debug_log_2026-03-15T093437.168.json). На этом уровне мы различаем не `Exception` в узком смысле, а более широкий класс нарушений контракта surface: analyzer называет их `lies`, а мы трактуем их как observable contract/invariant violations. Для текущего этапа это считается допустимым рабочим допущением: при отсутствии другого независимого измерителя выводы анализатора принимаются как истинные в границах данного прогона и данного набора скрытых стандартизированных запросов.

По подтверждённым данным текущего прогона установлено следующее: анализатор сообщил `197 API properties analyzed` и `39 corrupted` для window/non-worker части, а общий счётчик `totalLies` равен `75`. Разбор `lies` показал, что структура нарушений раскладывается на `43` corrupted API-entry: `4` worker API, у каждого из которых по `9` lie-меток, что даёт `36`, и ещё `39` non-worker API, у каждого из которых по одной lie-метке, что даёт ещё `39`. Таким образом, численное совпадение `75 = 36 + 39` подтверждено внешними артефактами, и для `Window scope` в данном прогоне все обнаруженные анализатором нарушения сведены к единственной метке `failed toString`, тогда как для `WorkerScope` это неверно, потому что там зафиксирован расширенный набор invariant-failures.

Из этого мы приняли рабочую контрольную точку: для `nav_butify`, в рамках именно этого эксперимента и именно этого анализа, window-scope surfaces не показывают других зафиксированных analyzer-нарушений, кроме `failed toString`; worker-scope surfaces остаются отдельно проблемным кластером с множественными контрактными сбоями; выпадения API из анализа или “не включившихся вообще” window API по данному отчёту не наблюдается. Все дальнейшие выводы должны опираться только на эти подтверждённые факты и не выходить за пределы того, что реально показано внешним анализатором в текущем прогоне.




В более позднем пайплайне bunch используется переработанная и более развитая версия механики, однако именно в этой ветке проявляется архитектурная несогласованность в той же области surface. В частности, те же API-точки — Navigator.webdriver и GPU.requestAdapter — по данным анализатора продолжают генерировать нарушения контракта, но уже иного типа, чем в предыдущем эксперименте.

Это делает сопоставление двух репозиториев удобным для сравнительного анализа: при одинаковых входных условиях и сопоставимом наборе скрытых запросов общее количество проверенных API и порядок величины повреждённых surface остаются практически неизменными (в пределах порядка 39–40 corrupted API, что для текущего наблюдения не является принципиальным расхождением). Однако структура lies отличается: набор фиксируемых invariant-нарушений в новой реализации распределяется иначе.

При этом важно, что в обоих случаях речь не идёт о выпадении API из анализа или о неинициализированных surface. Проверяемые API остаются доступными и участвуют в измерении, однако при обращении к ним наблюдаются различные типы контрактных сбоев. Именно поэтому данные два репозитория представляют удобную пару для структурного сравнения: при стабильных количественных метриках изменяется характер нарушений, что позволяет локализовать источник расхождения на уровне архитектуры реализации
